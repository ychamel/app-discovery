# OPEN_QUESTIONS — embeddable-update-widget

_Ambiguities, deferrals, and escalations. Any persona may add; resolutions are logged in
[DECISIONS.md](DECISIONS.md)._

| ID | Question | Raised by / stage | Status |
|----|----------|-------------------|--------|
| OQ-EUW-1 | Embedding mechanism — script-tag/iframe widget, a JSON/HTML endpoint the developer renders, or both? (Cross-origin + caching + zero-build-friction for the vibecoded-webapp niche.) | Coordinator seed / pre-1 | **RESOLVED (EUW-7, DESIGN §5.1 — pending DN-EUW-DESIGN):** a server-rendered `<iframe>` (one-line paste, no JS/build, cross-origin-safe by construction; `target="_top"` link → `/widget/<id>/view`). Script-injector + JSON-render-yourself rejected. |
| OQ-EUW-2 | **How** does a widget impression / click-through emit the non-curated D-7 **source** signal that AC9/EUW-6 now require — the exact `Surface` value (e.g. `WIDGET`), the emit shape, and how `developer-dashboard` reads it to show widget-attributed reach (M2/M3/M4)? Must stay **outside** `ratings.gate.CURATED_SURFACES` (D-8). | Coordinator seed / pre-1 | **RESOLVED (EUW-8/EUW-9, DESIGN §3/§6 — pending DN-EUW-DESIGN):** NOT a `signals` surface — a dedicated `apps/widget`-owned daily-rollup store (`widget_reach_count`); `apps/widget` imports nothing from `signals` ⇒ firewall **structural by absence** (stronger than "outside CURATED_SURFACES"). Dashboard reads it as a distinct off-platform slot. |
| OQ-EUW-3 | Anonymous serving — the widget renders to a developer's end users who are not platform accounts; what is read publicly, and what are the rate/abuse limits on an unauthenticated read surface? | Coordinator seed / pre-1 | **RESOLVED (DESIGN §5.2/§9 — pending DN-EUW-DESIGN):** public = the app's own notices + name + app-page link only (no private fields); per-IP fixed-window GET limit `widget_render_rate_limit_per_ip_per_minute()` (default 60) → 429; `Cache-Control` TTL bounds origin load. |
| OQ-EUW-4 | Is exposing a developer's published notices as a **public anonymous read** the intended product expansion (today they are follower-feed-only)? | Product Analyst / Stage 1 | **RESOLVED = yes (EUW-5, DN-EUW-BRIEF approved 2026-06-26)** — with the refinement that the view source is tracked for all users (EUW-6 / AC9). |
| OQ-EUW-5 | **Per-account conversion attribution (M3):** linking a *new account/follow* to the specific widget click-through that led to it — needs a widget-source token carried through an anonymous click → app page → sign-up (cookie consent + cross-domain identity + no-PII posture). | Software Architect / Stage 2 | **DEFERRED (EUW-10, DESIGN §11)** — out of MVP; AC9 reach (impressions + click-throughs) ships now. A token-carry design is a follow-up feature. |
