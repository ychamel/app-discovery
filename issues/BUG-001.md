### `BUG-001`: Duplicate interest subcategories click-target issue (label highlights previous occurrence)

- **Reporter:** Developer / QA Team
- **Date Reported:** 2026-06-28
- **Severity:** Medium
- **Status:** `RESOLVED`
- **Patch/Feature Slug:** `patch-interest-picker-duplicates` (Patch Track — core fix = unique per-cluster HTML IDs; the cross-cluster visual-sync enhancement was implemented following option 1 of `Q-001`)

#### Description & Impact
When choosing interests in the interest picker, tags that belong to multiple categories (clusters) are rendered multiple times (once under each category). Because the template generates the checkbox element ID using only the tag's UUID (`tag_{{ tag.id }}`), duplicate tags on the page end up sharing the same HTML ID. 

Consequently, when a user clicks the label of a tag in a subsequent category, the browser associates the label click with the first element in the DOM having that ID (the occurrence in the previous category). This toggles/highlights the checkbox in the previous category rather than the one the user hovered over and clicked, leading to a confusing and broken user experience.

#### Steps to Reproduce
1. Log in and navigate to the interests picker page `/interests/`.
2. Ensure there is a tag associated with multiple active clusters (e.g., tag "Python" in both the "Backend Development" and "Data Science" clusters).
3. Find the tag in the second cluster on the page.
4. Hover over and click on the text label for the tag in that second cluster.
5. Observe that the checkbox under the *first* cluster is checked/unchecked instead of the checkbox in the second cluster where the click occurred.

#### Expected Behavior
- Clicking a tag's label should toggle the checkbox next to it, not a checkbox under a different category.
- The checkbox DOM elements should have unique HTML IDs within the page (e.g., prefixing/suffixing with the cluster ID: `tag_{{ cluster.id }}_{{ tag.id }}`).
- If we want to keep duplicate tags visually separate, we should decide whether checking a tag in one category should visually sync and check the same tag in other categories on the page (via light progressive enhancement JavaScript), or if the page refresh after saving handles this.

#### Actual Behavior & Details
Clicking a duplicate tag label triggers the checkbox in the first category that has that tag.
In `apps/interests/templates/interests/picker.html`:
```html
<input type="checkbox" name="tag_id" id="tag_{{ item.tag.id }}" value="{{ item.tag.id }}"
       {% if item.checked %}checked{% endif %} style="width: auto; margin-top: 0.25rem;">
<label for="tag_{{ item.tag.id }}" ...>
```
If `item.tag.id` is not unique across the page, the HTML ID is duplicated, violating HTML standards and breaking label targeting.

#### Design Consideration: Duplicate vs. Unique Subcategories
We need to decide if we want to allow duplicate subcategories across different categories, or if all subcategories should be unique:

1. **Keep Duplicate Subcategories (Current Database Schema Support)**
   - *Pros:* Helps discovery. A topic like "Security" or "Python" naturally belongs to multiple areas (e.g., "DevOps" and "Backend"). Users can find interests through different pathways.
   - *Cons:* Visual redundancy. If a tag is selected, it should be visually synced across all categories in the UI to prevent the user from seeing "Python" checked in one box but unchecked in another on the same screen. This would require a small JS sync snippet.
2. **Enforce Unique Subcategories (Strict Parent-Child Taxonomy)**
   - *Pros:* Simpler UI/UX. Every interest subcategory appears exactly once on the page. No need for visual syncing.
   - *Cons:* Restricts tagging. Forces tags into a single category even when they span multiple disciplines, making them harder to discover.
