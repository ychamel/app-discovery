### `Q-002`: Should a user be able to submit a review for their own app?

- **Reporter:** User
- **Date Reported:** 2026-06-29
- **Category:** Product
- **Status:** `RESOLVED`
- **Patch Slug:** [`patch-block-self-interaction`](../patches/patch-block-self-interaction/) (bundled with `Q-003`)

#### The Question / Ambiguity
Currently, the app owner (the developer who submitted an app) can navigate to their own app's detail page and submit a star-rating / written review — the same as any other user. Is this the intended behaviour?

#### Context & Alternatives
Self-reviews are a known abuse vector on every app marketplace (App Store, Google Play, Product Hunt). Allowing them:
- Inflates the app's visible rating immediately after launch, before any real user reviews arrive.
- Undermines reviewer credibility: the developer has an obvious conflict of interest.

**Option A — Block self-review (recommended):** Hide or disable the review form when `request.user == app.developer`. Show a non-dismissible notice: *"You can't review your own app."*

**Option B — Allow but flag:** Accept the review but visually mark it as "by the developer" and exclude it from the aggregate score calculation.

**Option C — Keep as-is:** No restriction. Simplest to implement; relies on community flagging.

#### Proposed Resolution / Answer
**Answered 2026-06-29 → Option A (block self-review)** via DN-Q002. Hide/disable the review
form when `request.user == app.owner` *and* reject the rating mutation server-side (the
template hide alone is bypassable — CLAUDE.md §5.4). Routed to the Patch Track as
[`patch-block-self-interaction`](../patches/patch-block-self-interaction/), bundled with
`Q-003` (symmetrical owner-guard). No schema change — owner identity is already on the
`CatalogApp` read-model.
