### `Q-001`: Duplicate vs. Unique Interest Subcategories Design Choice

- **Reporter:** Developer / QA Team
- **Date Reported:** 2026-06-28
- **Category:** Product / Strategy
- **Status:** `NEW`

#### The Question / Ambiguity
Should the taxonomy model and interest picker interface allow subcategories (tags) to exist under multiple categories (clusters) as duplicates, or should we enforce a strict single-parent (unique) taxonomy?

#### Context & Alternatives
This question was surfaced during the investigation of `BUG-001`, where tag click labels highlighted duplicate occurrences in previous clusters because they shared duplicate HTML IDs. 

There are two primary alternatives:

##### Option 1: Keep Duplicate Subcategories (Multi-Cluster Mapping)
- **Description:** A tag like "Python" can belong to both "Backend Development" and "Data Science" clusters.
- **Pros:** Matches real-world classification; users can find tags via multiple pathways.
- **Cons:** Visual redundancy. Requires frontend JavaScript/HTMX sync to ensure selecting "Python" in one cluster instantly checks it in all other clusters on the same screen to prevent visual drift.

##### Option 2: Enforce Unique Subcategories (Strict Taxonomy hierarchy)
- **Description:** Each tag can belong to at most one cluster.
- **Pros:** Simpler UI/UX. No duplicate elements on the page, meaning no ID collisions and no need for visual syncing.
- **Cons:** Restrictive. Forces a binary choice for multi-disciplinary tags.
