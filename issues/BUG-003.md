### `BUG-003`: Selected reporting period shows literal `{{ w.label }}` in the developer dashboard

- **Reporter:** User
- **Date Reported:** 2026-06-28
- **Severity:** Medium
- **Status:** `RESOLVED` (2026-06-29 — closed-out, released local/dev)
- **Patch/Feature Slug:** `patch-dashboard-window-label` (Patch Track — template-only one-line fix)

> **RESOLVED.** Both multiline variable tags in `my_apps.html` were joined onto one line:
> the reported selected-window `{{ w.label }}` **and** a sibling
> `{{ summary.curated_impressions }}` surfaced by the repo-wide sweep (it leaked literal
> text whenever a developer had curated impressions). A red-first regression test
> (`test_template_tags_render_no_literal_braces`) guards both plus any future multiline
> tag in this view. 999 tests green; ruff/check clean; no migration drift; the sweep now
> returns zero matches across all templates. See
> [RELEASE_NOTES.md](../patches/patch-dashboard-window-label/RELEASE_NOTES.md).

#### Description & Impact
In the developer dashboard, the **Reporting Period** selector renders the literal text
`{{ w.label }}` for the *currently selected* window instead of its human label (e.g.
"Last 7 days"). The non-selected windows render correctly. This makes the active period
look broken/unstyled and hides which period is in effect.

#### Steps to Reproduce
1. Log in as a developer and open the dashboard "My apps" page (`apps/dashboard/`,
   [`my_apps.html`](../apps/dashboard/templates/dashboard/my_apps.html)).
2. Look at the **Reporting Period** segmented control at the top.
3. Observe the **selected** (highlighted) period button shows the raw text
   `{{ w.label }}` rather than its label; the other (clickable) periods are fine.

#### Expected Behavior
The selected period button shows its human-readable label (the same `w.label` the other
buttons show), just rendered in the active/primary style.

#### Actual Behavior & Details
Root cause is **certain** and template-only. In
[`apps/dashboard/templates/dashboard/my_apps.html`](../apps/dashboard/templates/dashboard/my_apps.html#L32-L33)
the active-window branch splits the Django variable tag across two lines:

```html
<span aria-current="true" class="btn btn--primary btn--sm" ...>{{
  w.label }}</span>
```

Django's template tokenizer (`django.template.base.tag_re`,
`({%.*?%}|{{.*?}}|{#.*?#})`) is compiled **without** `re.DOTALL`, so `.` does not match
the newline — a `{{ … }}` variable tag that spans two lines is **not** recognized and is
emitted as literal text. Verified empirically on Django 5.2.15:
`tag_re.findall("{{\\n w.label }}")` returns nothing. The non-selected branch
([:36](../apps/dashboard/templates/dashboard/my_apps.html#L36)) keeps `{{ w.label }}` on
one line, which is why only the *selected* period misrenders.

**Fix:** put the variable tag on a single line: `…>{{ w.label }}</span>`.

A repo-wide check for the same multiline-tag pattern in other templates is in scope for
the patch (cheap insurance against the same class of bug elsewhere).
