### `BUG-005`: "Share" button does not work on localhost

- **Reporter:** User
- **Date Reported:** 2026-06-29
- **Severity:** Low (localhost) / Medium (production risk)
- **Status:** `NEW`
- **Patch/Feature Slug:** `TBD`

#### Description & Impact
The **"Share"** button on the app detail page does not produce the expected behaviour when the app is running on `localhost`. It is unclear whether this is a known browser restriction or a code-level bug that will also manifest in production.

#### Steps to Reproduce
1. Run the platform locally (`localhost:<port>`).
2. Open any app detail page.
3. Click the **"Share"** button.
4. Observe: nothing happens, an error appears in the browser console, or the share dialog does not open.

#### Expected Behavior
Clicking "Share" should either:
- Open the browser's native Web Share API dialog (`navigator.share()`), allowing the user to share the app URL via OS-level share targets, **or**
- Copy the canonical app URL to the clipboard and show a confirmation toast, **or**
- Open a share sheet/modal with platform-specific share links (Twitter/X, copy-link, etc.).

#### Actual Behavior & Details
The share action fails silently or throws an error. Two likely root causes to investigate:

1. **`navigator.share()` requires a secure context (HTTPS).** On `localhost` without SSL, `navigator.share` is `undefined` in some browsers. This is a known browser security restriction — the fix is to add a fallback (e.g., clipboard copy) when the Web Share API is unavailable.

2. **`navigator.share()` is not supported in all desktop browsers** (notably Firefox desktop as of mid-2025). A fallback is required regardless of environment.

**Action required:**
- Check the browser console for errors when clicking Share on localhost.
- Verify whether the Share button works correctly on a staging/HTTPS URL.
- If the issue persists on HTTPS, it is a code bug. If it only fails on HTTP localhost, implement the clipboard-copy fallback for non-secure/non-supported contexts.
