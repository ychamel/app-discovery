# DECISIONS.md — Global Decision Log (ADRs)

**This is the durable channel for repo-wide rationale.** It records decisions that
outlive any single feature — the ones a future feature's Architect must not have to
re-derive or accidentally contradict.

## Global vs. per-feature decisions

- **Here (global):** the tech stack, the shared-code root, the ranking algorithm, the
  beachhead niche, data-store choice, auth model, cross-cutting conventions — anything
  that constrains *more than one* feature.
- **In `features/<slug>/DECISIONS.md` (local):** choices scoped to one feature that no
  other feature needs to know about.

Rule of thumb: **if a later feature would be wrong to contradict it, it belongs here.**
A global decision buried in one feature's folder is invisible to the next feature — that
is exactly the drift this file prevents.

## Format (one ADR per entry, newest first)

```
### D-<n>: <short title>
- **Date:** YYYY-MM-DD
- **Stage / feature:** <where it was made>
- **Decision:** <what was chosen>
- **Why:** <the reasoning>
- **Alternatives rejected:** <≥1 genuinely different option + why not>
- **Sacrifices / consequences:** <what this costs us>
```

A confirmed global decision is also summarized in [CONTROL.md](CONTROL.md)
*Decisions Made (recently)* as a one-line, human-readable digest.

## Decisions

### D-13: Frontend premium look is delivered build-free — a hand-authored, token-driven CSS design system; no Tailwind/Node toolchain (Tailwind kept as an outcome-gated, reversible fallback)
- **Date:** 2026-06-28
- **Stage / feature:** `2-design` / `premium-frontend` (Software Architect) — **APPROVED** (resolves **DN-PF-DESIGN**; promotes feature-local PF-DESIGN-1…7 → RATIFIED). **Affirms and extends [D-12](DECISIONS.md)** (its build-free, WhiteNoise-served stylesheet posture is **preserved, not revised**) inside the [D-4](DECISIONS.md) server-rendered envelope. Resolves the brief's deferred **PF-2 / C3** ("Tailwind's build step"). Full design in [features/premium-frontend/DESIGN.md](features/premium-frontend/DESIGN.md).
- **Decision:** The platform's premium frontend is delivered **without a CSS build step**. The single, hand-authored `apps/core/static/core/app.css` (one source of truth, served + hashed by WhiteNoise) is **deepened into a full design system**: all colour/size/space literals live only in a `:root` **token** block (palette + states + elevation, a modular **type scale**, a **spacing scale**, motion gated by `prefers-reduced-motion`); above it sit reusable **layout primitives + a styled component set**, organised as clearly-labelled sections in the one file. **No Tailwind, no Node/npm, no build step** is introduced; the `render.yaml` `buildCommand` stays pure-Python (`pip install` + `collectstatic`). Light interactivity uses **HTMX vendored + self-hosted** (`apps/core/static/core/vendor/htmx.min.js`, no CDN), loaded `defer`, applied only as `hx-boost` progressive enhancement that **degrades to the no-JS path** (the no-JS path is the source of truth). The widget firewall (third-party iframe templates link/extend nothing from `core`) is preserved by construction. **No new model, schema, or migration.**
- **Why:** The premium gap is **design depth, not tooling** — the repo already had a working hand-authored design system (tokens + components); it was shallow, not mis-architected, and Tailwind does not *give* a premium look (you still design the tokens/scale; it changes *where* they live — utility classes in markup vs. one component stylesheet). Build-free **eliminates the brief's #1 risk R1** (a build step destabilising the just-stabilised, parked [D-12](DECISIONS.md) staging deploy), keeps [D-4](DECISIONS.md)'s single-language (Python) repo intact, and best serves the brief's AC-5 ("one design system, no per-page ad-hoc styling, one source of truth"). This is global because it settles the **build posture every future surface inherits** (a later feature would be wrong to silently add a Node toolchain). CLAUDE.md §5.5 (simplicity-first, boring-and-well-understood, no speculative abstraction, match-existing-conventions) points the same way.
- **Reversibility / revisit trigger (user, 2026-06-28):** chosen **"for now," to observe the outcome.** The outcome to watch is the **M7 premium sign-off** (and future authoring velocity once the design system spans more surfaces). If the hand-authored approach **cannot reach the premium bar** or proves too slow to extend, **revisit → adopt Tailwind via Option B** (see rejected alternatives). The migration is designed to be clean: the `:root` token system maps directly onto a Tailwind config, and Option B reuses the **same** single served/hashed `app.css` contract + the widget firewall, so the swap is additive and does not disturb the D-12 serving model. This is a flagged, expected-to-be-cheap reversal, not a one-way door.
- **Alternatives rejected:** (a) **Tailwind via the standalone CLI binary** (no Node/npm; one `tailwindcss -i src.css -o app.css --minify` build step before `collectstatic`, binary fetched/vendored, `content` glob over `apps/**/templates/**`) — the **reversible fallback** above, *fully specified in [DESIGN.md](features/premium-frontend/DESIGN.md) §11* so it is buildable on the revisit trigger; rejected **for now** because it reintroduces a build step (R1 at lower severity — a binary fetch can fail a deploy) and pushes styling into template markup, for authoring-velocity benefit not needed at the current surface count. (b) **Tailwind via the full Node/npm toolchain** (`package.json` + `node_modules` + a Node runtime) — rejected: heaviest build into a pure-Python repo (D-4) and the just-stabilised deploy (R1 at full severity). (c) **Tailwind Play CDN** (runtime browser JIT) — rejected outright: makes the whole stylesheet depend on JS + an external origin → breaks no-JS (AC-1), regresses first paint (M2), adds a supply-chain/privacy dependency, explicitly not for production. (d) **A self-hosted web font** — deferred (a render-blocking/FOUT request risks M2; a refined system-font stack delivers the look at zero first-paint cost; revisit only if M7 flags typography).
- **Sacrifices / consequences:** We hand-author the type/spacing scale and components rather than inheriting Tailwind's ready-made defaults — a one-time design cost, paid once then reused; and we forgo utility-class authoring velocity. Accepted because it is paid once and the reversal path (above) is cheap if that proves wrong. The design system (the `:root` token names + the component class API) is now the **stable, additive-only styling substrate** every later surface inherits (Q1 authed-surface restyle is a cheap follow-on); renaming a token is a single-file change, but the **token names are a cross-surface contract** — change them additively. Rollback of the *feature* (not the decision) is a single `git revert` of the build commit (CSS + templates + the one landing route + vendored JS come out together, no data migration — the DU-REL-1 pattern).

### D-12: Deployment / serving stack + shared frontend shell — Render Blueprint, gunicorn/WhiteNoise, Resend SMTP, Redis-backed cache, a DB-only liveness probe, and one consolidated responsive base template
- **Date:** 2026-06-27
- **Stage / feature:** `2-design` / `platform-staging` (Software Architect) — **APPROVED** (resolves **DN-PS-DESIGN**; promotes feature-local PS-DESIGN-1…8 → RATIFIED). Executes [D-11](DECISIONS.md) (stand up reachable staging) and **implements** [D-4](DECISIONS.md)'s "email provider is ops/env config" + server-rendered-templates clauses; introduces no SPA and no second language. Bounded by **PS-1** (free tier now, $20–100/mo durable, prod-bound). Full design in [features/platform-staging/DESIGN.md](features/platform-staging/DESIGN.md).
- **Decision:** The platform's deployment + serving stack and frontend shell are fixed as: **(1) Host = Render** via a declarative **Blueprint** (`render.yaml`: one web service + managed PostgreSQL + a persistent disk + free Redis); a `*.onrender.com` HTTPS URL serves staging, a custom domain is an env-only prod-promotion step. **(2) WSGI/static/DB = gunicorn** serving `config.wsgi` + **WhiteNoise** (`CompressedManifestStaticFilesStorage`, serves app *and* Django-admin assets from `STATIC_ROOT`, no second web server) + **`dj-database-url`** (`DATABASE_URL` with the discrete `DB_*` vars as the local fallback). The release step is `migrate --noinput && collectstatic --noinput`. **(3) Media** lives on a **persistent disk** at `MEDIA_ROOT` with a non-`DEBUG` route; an S3-compatible **object store** (Cloudflare R2 + `django-storages`, a `STORAGES["default"]`-only swap with no caller change) is the documented 100×/prod growth path, not built. **(4) Email = Resend** over Django's SMTP backend — **pure config** (`apps/core/email.py` already swaps by `EMAIL_BACKEND`); ships `docs/deploy/email-provider-setup.md`; Postmark is the deliverability-premium alternative. **(5) Cache** = Django 5's built-in `RedisCache` wired from a `REDIS_URL` env var with a **LocMemCache fallback when unset** (local/test unchanged), so the fail-open auth rate limiter (`apps/core/ratelimit.py`) is correct across gunicorn workers. **(6) Observability** = existing structured stdout logs + **env-gated Sentry** (`SENTRY_DSN` unset = disabled) + a **new DB-only `GET /health/live`** liveness probe split from the existing `/health` (which opens a live SMTP socket and is therefore unsafe as an orchestrator target). **(7) Frontend** = **one shared responsive base template + one dependency-free, build-free stylesheet** in `apps/core` (`core/base.html` + `core/static/core/app.css`), with the **6 duplicated per-app `base.html` consolidated** onto it; the embeddable **widget stays isolated** (it must not pull the platform stylesheet — preserves the AC3.3 firewall). New deps in [pyproject.toml](pyproject.toml): `gunicorn`, `whitenoise`, `dj-database-url`, `redis`, `sentry-sdk[django]`. **No schema/migration.** Every new behavior is **env-gated with a dev-safe default**, so local dev and the test suite are unaffected and rollback = tear the Blueprint down.
- **Why:** Global because it binds infrastructure and a frontend shell every later feature inherits — a future feature would be wrong to hardcode DB config, bypass the shared shell, or assume a different host/email/cache wiring. The choices are the **boring, well-understood, config-replaceable** ones (CLAUDE.md §5.5, §5.2): the 12-factor posture means host/DB/cache/email each swap by env, not code; Render's repeatability is **structural** (the Blueprint *is* the second-run reproduction, satisfying the AC2.1 "no undocumented manual fix-ups" gate) rather than a runbook of manual clicks; WhiteNoise + `dj-database-url` close the real C4 static gap (no `STATIC_ROOT`/0 static files today) with zero extra services; the Redis-backed cache fixes a genuine multi-worker security degradation (per-worker LocMem counters relax the limiter N×); the liveness split removes a real production footgun found in code; and consolidating six near-identical base templates with no CSS is the only non-duplicative way to do the mandated responsive polish (CLAUDE.md §5.1/§5.3, design-for-deletion).
- **Alternatives rejected:** **Host —** Heroku (no free tier; Procfile path noted as the portability note), Fly.io (more ops surface — volumes/regions — for no staging benefit), a raw VPS (hand-rolled nginx/gunicorn/certbot/systemd = exactly the undocumented manual surface AC2.1 forbids). **Static —** a CDN/nginx sidecar (an extra service; a CDN in front of WhiteNoise is the env/DNS-only growth path). **Media —** object store now (a speculative dep/setup before it's needed, §5.5). **Cache —** single-worker gunicorn (also makes the limiter correct, but caps throughput; kept as the fallback if managed Redis is unavailable). **Email —** Postmark now (deliverability-premium, $15/mo; the documented upgrade if inbox placement becomes the bottleneck). **Frontend —** polishing the 6 `base.html` in place (faster, but duplicates the stylesheet six times = debt violating §5.1/§5.3) and a dedicated SPA (deferred + evidence-gated by D-11). Render's lock-in is mitigated because every choice under it is standard Django + a single declarative file.
- **Sacrifices / consequences:** A small amount of host lock-in (mitigated by standard Django + one declarative file) and single-node media/throughput at staging (both with named, config-only growth paths: R2 object store, a CDN, more workers). Cost moves from $0 (free tier) to ~$14–25/mo on the durable/prod-bound tier (inside PS-1's $20–100). The frontend consolidation is a one-time refactor touching ~30 templates (mechanical, sequenced in Stage 3) plus a render-every-surface check. The two `.md` deliverables (deploy runbook, email-provider setup) become part of the committed deploy artifacts. AC3.1/M5 (magic-link delivery) still gate on a real inbox once the provider is wired (PS-2/PS-3). The contract is additive (a liveness route + new template blocks) — no existing consumer changes.

### D-11: Validate the wedge in staging (full-role, web+mobile) before live recruitment; keep server-rendered Django templates until the walkthrough proves a surface needs more
- **Date:** 2026-06-27
- **Stage / feature:** Off-pipeline **Strategist** (CLAUDE.md §4.1), ratified by the user (resolves **DN-STRAT-1** + the frontend sub-call). **Refines** the [D-10](DECISIONS.md) wedge sequence (adds a validation gate *before* §5.4-step-3 recruitment); **affirms** (does not supersede) [D-4](DECISIONS.md)'s server-rendered-templates choice and its "richer SPA deferred until a surface needs one" clause. Not yet a feature folder — the next Coordinator (§4) scope.
- **Decision:** The next bet is **not another product feature and not yet a public launch** — it is to **stand up a reachable staging environment, deploy the already-built wedge to it, and run a full end-to-end UX walkthrough across all three roles (user / developer / admin) on both web and mobile** before recruiting any real founding developers. The walkthrough is the **forcing function** for two things: (a) it finally exercises the deferred deployment work (hosting target, domain, the [D-4](DECISIONS.md) email provider currently stubbed, basic monitoring), and (b) it produces the **evidence** that decides the frontend architecture. On the frontend: **keep the server-rendered Django templates** for the wedge and invest in **polishing them + making them mobile-responsive**; a split to a dedicated SPA frontend (Django-as-backend-API) is **deferred and evidence-gated** — undertaken only if the staging walkthrough shows a specific surface the templates demonstrably cannot serve, at which point it is an explicit D-4-revisit ADR, not a silent rewrite. Live recruitment (§5.4 step 3) follows *after* staging validation passes; the network ([D-10](DECISIONS.md)), monetization ([D-9](DECISIONS.md)), and editorial-curation remain **density-gated behind real adoption** and unchanged.
- **Why:** 13 features are built but **all released to local/dev only — nothing has ever touched a real developer or user**, so H1/H2/H3 and the §8 one-line test are unvalidated, and every option on the standing fork (D-9 / network / editorial) is **density-gated and therefore premature** (none can be un-gated without real adoption, which requires being reachable). You cannot responsibly put founding developers onto a UI that has never been exercised end-to-end or checked on mobile — and the wedge's load-bearing surfaces (the public **app page** + the **embeddable widget**) *are the developer's public face*, so their UX quality is load-bearing for the bring-your-own-audience thesis itself. Deciding the frontend architecture *before* that walkthrough would be a speculative multi-week investment (CLAUDE.md §5.5) made on a hunch; server-rendered templates are also the better fit for content-heavy, SEO-sensitive public pages and a zero-build embed (an SPA would need SSR just to match). This is global because it re-orders what the next feature is and binds the frontend posture every later wedge surface inherits.
- **Alternatives rejected:** (a) **Go straight to public launch + recruitment** (the Strategist's first recommendation) — eager; recruits real developers onto an un-walked, mobile-unchecked UI, risking the wedge's public face on contact. (b) **Build a fourth density-gated feature first** (D-9 / network / editorial) — the "keep building, never ship" trap; none can be validated without adoption. (c) **Commit now to a dedicated SPA frontend** — a large speculative cost (second toolchain, build pipeline, auth/session bridging, SSR for crawlable app pages) before any evidence the templates are insufficient; violates §5.5 and D-4's explicit deferral. (d) **Decide the frontend with no walkthrough at all** — would either gold-plate or under-build blind; the walkthrough is the cheap evidence source.
- **Sacrifices / consequences:** Adds a staging/validation step before any revenue or adoption — a deliberately *longer* path to a first real user, accepted as the honest cost of not burning the founding cohort on an unvalidated UI. The frontend question stays **open but bounded**: the working assumption is "templates, polished + responsive," and a rewrite is a reviewed D-4-revisit triggered only by walkthrough evidence (no silent drift). Deployment specifics (hosting, domain, email provider, monitoring) are this next feature's Stage-1/2 concern, not pre-decided here (consistent with [D-2](DECISIONS.md)). Density-gated bets (D-9, network, editorial) are untouched and still wait for adoption.

### D-10: MVP cold-start re-sequenced — lead with a single-player developer hub; gate the discovery network on density
- **Date:** 2026-06-26
- **Stage / feature:** `0-coordinator` (user-directed strategic pivot). Activates the new feature `embeddable-update-widget`; deprioritizes `editorial-curation-tools` + `weekly-digest` to backlog.
- **Decision:** The MVP wedge is a **developer hub**, not editorial curation. Lead with **single-player developer value** — the easiest way to spin up a beautiful public **app page** + an **update/changelog feed** + an **embeddable "what's new" widget** the developer drops inside their own app — useful to a developer with **zero followers on day one**, so that each adopter **brings their own existing audience onto the platform** (the bring-your-own-audience engine). **Hold the network/discovery features back until per-niche density** (the personalized digest, trending, cross-app browse beyond the open search surface): worthless at ten apps, powerful at a few hundred in one niche. This **re-sequences vision §5.4** (which led user-side with the digest) to **developer-wedge-first**. It does **not** change the beachhead niche ([D-1](DECISIONS.md), vibecoded webapps) or the impression-economy / Quality-Score / ring design — those are unchanged and turn on at density. Most of the wedge is **already built** (`app-pages`, `developer-updates`, `app-subscriptions`, `developer-dashboard`, `open-search-browse`); the missing, load-bearing piece is the **embeddable widget** (the user-capture mechanism a shared link alone doesn't provide), which becomes the next feature.
- **Why:** Solve the chicken-and-egg by **not having one** — give the hard side (developers, who create the content users consume) standalone value on an empty platform, and let them drag their audiences in as a side effect. A digest-first or two-sided-simultaneous launch leaves the platform empty and useless to both sides until a critical mass that neither side will wait for. This is global because it re-orders the launch sequence every Phase-2+ feature is scheduled against and changes which features are active vs. deferred — a later feature would be wrong to assume the digest-first order.
- **Alternatives rejected:** (a) **Digest-first** (the original §5.4) — the user side has nothing to open with ten apps; scarcity can't substitute for an empty catalog, and it acquires no developers. (b) **Two-sided simultaneous launch** — the canonical chicken-and-egg; both sides churn before density. (c) **Keep editorial-curation-tools as the MVP** — curation/matching is a network feature; it has no value before there are users *and* apps to match, so it is correctly held until density.
- **Sacrifices / consequences:** **No moat during the wedge phase** — a standalone changelog/landing-page tool is a commodity (Beamer/Canny/changelog.com exist); defensibility (aggregation into discovery) only switches on at density. **`editorial-curation-tools` + `weekly-digest` move to backlog** until the network turns on (reactivation = a future D2 call). Combined with [D-9](DECISIONS.md) (free tools), **the wedge phase has no monetization** — a longer runway-to-revenue, consciously accepted. The niche (D-1) and ranking design are untouched, so nothing already built is invalidated.

### D-9: Monetization model — paid, labeled, time-boxed promotion placements firewalled from the ranking; tools and curation stay free
- **Date:** 2026-06-26
- **Stage / feature:** `0-coordinator` (user-directed). **Supersedes** the earlier per-app developer *tool*-subscription sketch (recorded under DN-20, 2026-06-24). Not yet a feature — no folder/Stage (density-gated by [D-10](DECISIONS.md)).
- **Decision:** Revenue comes from selling **attention, never ranking**. A developer pays for a **clearly-labeled, time-boxed promotion placement** ("marketing" section) to push an announcement — a **new release**, a **major update**, or a **beta-tester recruitment** call — gaining reach for a bounded period. **Tools** (dashboard, the update/changelog channel, the app page) **and curation** (the earned discovery surfaces) are **free** — paying is never required to compete. Promotion placements are a **non-curated surface**: by construction they **cannot move the Quality Score**, because they sit **outside `ratings.gate.CURATED_SURFACES`** ([D-8](DECISIONS.md)) — i.e. a new `Surface` value ([D-7](DECISIONS.md)) deliberately kept out of the curated set. The firewall is **per-impression, not per-user**: a promo impression never counts, but a user *acquired* via a promo who later gets the app **organically curated** and rates it **does** count. **Guardrail (lock before revenue exists):** promotion placements are **visually distinct, always labeled, and bounded as a published fraction of the feed**.
- **Why:** The platform's premise is that **money can't buy position** (vision §2/§4/§5.6/§8). Promos buy **top-of-funnel attention**, never **ranking** — and the existing D-7/D-8 architecture makes that true *by construction* (non-curated surfaces are already firewalled from the score), so this monetization is coherent with the integrity model rather than bolted onto it. It also aligns with §4.2's equilibrium ("manipulation should approach the cost of just acquiring real users"): the platform now **sells that legitimate path**. Global because the firewall (and the `Surface`-outside-`CURATED_SURFACES` rule) binds any future promotion feature and any score consumer.
- **Alternatives rejected:** (a) **Per-app tool subscription** (the prior sketch) — taxes a commodity tool developers can get free elsewhere (weak willingness-to-pay, esp. in the indie/build-in-public niche) and makes "free to launch and grow" harder to honor. (b) **Paid ranking / position** — forbidden by the premise; the one thing the platform must never sell. (c) **Promotion inventory with no labeled cap** — the erosion path every "fair" platform dies by (ads grow, get more prominent, start to look like curation — §5.6 names this failure); a published, locked cap is the structural defense.
- **Sacrifices / consequences:** The promise **narrows honestly** — money now **can** buy *visibility* (top-of-funnel), so "money can't buy visibility" becomes "**money can't buy ranking**"; this must be **published as a trust feature** (same ethos as §5.6's "publish the ranking-inputs list"). Revenue is **density-gated** — promotions only sell once there's an aggregated audience to reach, so the [D-10](DECISIONS.md) wedge phase earns nothing (accepted). Not yet built; scaffolding the feature is a future Coordinator/D2 call.

### D-8: The curated-rating gate semantic — "curated to app X" = an organic-curation (DIGEST) impression
- **Date:** 2026-06-20
- **Stage / feature:** `2-design` / `ratings-reviews` (Software Architect) — **APPROVED 2026-06-21** (DN-12 → approve; DESIGN.md ratified, finalized on hand-off to Stage 3; promotes the feature-local [RR-2](features/ratings-reviews/DECISIONS.md) to repo-wide). Implemented in Stage 4 (`apps/ratings`). **Rollout:** any feature that *produces* curation or *consumes* the gate must agree with this definition before it ships.
- **Decision:** A user is **"curated to app X"** — and therefore a rating they leave on X is **weight-eligible** — **iff they have a [D-7](DECISIONS.md) `Impression` of X on an *organic-curation* surface**. The set of organic-curation surfaces is **`{Surface.DIGEST}`** today; a `Surface.APP_PAGE` (open/direct page view) or any self-driven view **never** confers eligibility (vision §4.1: "only users to whom the app was *organically curated* may affect its score"). The gate governs a rating's **weight**, never whether it may be **posted** (anyone signed-in may rate; outside ratings are displayed-but-unweighted — RR-3). The determination is **recorded data, not a computed verdict** (RR-1): the layer that captures it performs **no** scoring; the curated-surface set is the single source of "what counts as curation" and lives in one place (`apps.ratings.gate.CURATED_SURFACES`); the read of impression evidence goes through the D-7 selector surface (`signals.selectors.has_impression`), never a direct `signals_*` read.
- **Why:** This is the platform's single most important integrity rule (vision §4.1 — "the rule that kills the cheapest, most common attack" of bought/farmed ratings). More than one feature depends on a **shared** definition of it: **`editorial-curation-tools`** must *produce* the impressions that make a user curated; **`developer-dashboard`** reports "reach = curated users"; and the **Quality Score** consumer weights only eligible ratings. If any of them defined "curated" differently the gate would be inconsistent across the platform and the §4.1 premise would leak. Per this file's rule of thumb — *"if a later feature would be wrong to contradict it, it belongs here"* — the gate semantic is global. Anchoring it on a `DIGEST` impression keeps it **forward-compatible with no schema change** (it reads the existing D-7 `Surface` enum) and **honest at MVP**: with no `DIGEST` emitter yet, ~all ratings record *not-eligible* (correct, made visible by the gate-split metric), becoming meaningful automatically when `weekly-digest` / `editorial-curation-tools` emit `DIGEST` impressions.
- **Alternatives rejected:** (a) **Any platform impression counts (incl. `APP_PAGE`)** — an open/self-driven page view is not *organic curation*; it would let a user confer eligibility on themselves by visiting, defeating §4.1. (b) **Record raw evidence and defer the verdict entirely to the score consumer** — more deferral than needed once the rule is this clear; the raw D-7 impressions persist anyway, so a re-derivation path already remains. (c) **Editorial assignment only** (a human editor assigned X to the user) — couples the gate to the unbuilt `editorial-curation-tools`; a `DIGEST` impression is the surface-level proxy *any* curation mechanism produces, so the gate can be defined on it now and editorial assignment becomes one more emitter of `DIGEST` impressions. (d) **Keep the definition feature-local to `ratings-reviews`** — invisible to the next feature that must agree with it (the exact drift this file prevents). All rejected.
- **Sacrifices / consequences:** At MVP the gate is ~always *not-eligible* (no `DIGEST` emitter — `ratings-reviews` R3); this is correct, not inert, and observable via the gate-split metric. Evolving the gate (e.g. adding an editorial-assignment surface, or a "feed" surface) is a deliberate one-line change to `CURATED_SURFACES` plus this ADR — additive and reviewed, never silent. The determination is *frozen per rating* (so the score has a 100%-present field — `ratings-reviews` AC5) but *re-derivable* from the retained inputs + the append-only corpus, so a definition change is correctable by recompute rather than a destructive migration. The gate reads, but does **not** produce, curation — the curated surface is owned by `weekly-digest`/`editorial-curation-tools`.

### D-7: Behavioral event-schema contract — append-only events, captured through one write path, read raw
- **Date:** 2026-06-18
- **Stage / feature:** `2-design` / `signal-capture` (Software Architect) — **APPROVED 2026-06-18** (DN-6 → approved; DESIGN.md ratified, finalized on hand-off to Stage 3). Implemented in Stage 4 (`apps/signals`). **Rollout ([DESIGN §11/§12](features/signal-capture/DESIGN.md)):** an emitting/consuming feature must adopt these rules *before* it emits or reads any behavioral signal.
- **Decision:** A behavioral signal is an **append-only event** (written once, never mutated, never deleted) in one of **two shapes**: an **`Impression`** — the anchor, with its **own UUID identity**, keyed `user × App.id`, a `surface`, and a **frozen capture-time `Tag.id` snapshot** of the app's categories *as shown*; or an **`EngagementEvent`** — `kind ∈ {click_through, subscribe, page_reengagement, share, off_platform_proxy}` (a closed code enum), keyed `user × App.id`, with an **optional `impression` link** (required for `click_through`/`off_platform_proxy`) and an `is_proxy` flag. Every emitting feature (**`weekly-digest`, `app-pages`, `app-subscriptions`, `developer-updates`**) must: **emit only through `signals.capture.*`** (the single write path that validates the app via D-6 `get_catalogued_app`, snapshots tags, links the impression, counts the write, and **fails loud** — no surface writes `signals_*` directly); **reference apps by `App.id` (D-6) and tags by `Tag.id` (D-5)** as soft UUID refs, never by label/URL or a hard FK; and treat the corpus as **raw** — **all scoring/normalization is a downstream consumer's job, never done in this layer** (the schema has **no score/weight/rank column**). **Return-to-platform @3d/@14d is *derived* at read** (impression × `PlatformVisit` activity), **not a stored event** — so "did-not-return" (an absence) is representable and no backfill/scheduled job exists. The **off-platform proxy** is a **flagged secondary** signal (`is_proxy=True`), never required for funnel completeness. The published read surface is **raw counts only** via `signals.selectors.*`.
- **Why:** This is the schema-first spine of the whole platform (breakdown §4.5): the Quality Score, ring expansion, the impression allocator, and the integrity graph are all later **consumers** of this corpus. Modeled well now (per-impression identity, `user × App.id × impression` keys, capture-time category tags, raw-not-scored) those systems are consumers, not rewrites; modeled badly the north-star architecture inherits near-irreversible debt (signal-capture R2). Routing all writes through `capture.*` puts the keys, the app/tag validation, the fail-loud rule, and the completeness metric in **one** enforced place rather than re-implemented per surface. Append-only + raw keeps the corpus a trustworthy backtest substrate (H3): facts don't drift and no premature scoring contaminates it. This is global because **more than one later feature would be wrong to contradict it.**
- **Alternatives rejected:** (a) **A single wide polymorphic event table** (one table, nullable tag/window/impression columns + a type tag) — makes illegal states representable (a share with a return window, an impression with no tags) and hides the impression's anchor role; rejected for an impression-anchor table + a uniform engagement table. (b) **Table-per-event-type** (5–6 tables) — the downstream kinds share an identical column shape, so it adds tables + UNION reads for zero benefit and turns "add a kind" into "add a table"; rejected for one `EngagementEvent` with a `kind` enum. (c) **Materialize returns in a cohort table via a scheduled job** — invents a "did-not-return" representation problem, needs batch infra, and risks drift vs source rows; rejected for backfill-free read-time derivation. (d) **Let each surface write events ad hoc** — duplicates keys/validation/fail-loud per surface (the exact drift this contract prevents). Full rationale in [features/signal-capture/DESIGN.md](features/signal-capture/DESIGN.md) §6/§13.
- **Sacrifices / consequences:** No first-class stored "return" row (derived at read — a bounded read cost; a materialized projection is the documented 100× growth path). No DB referential integrity on `app_id`/`tag_id` (boundary validation + soft refs instead, inherited from D-5/D-6 — a later app withdrawal deliberately does **not** cascade-erase historical signal). Behavioral facts about a **deleted** account are **retained but anonymized** (`user` FK `SET_NULL`; `PlatformVisit` `CASCADE`) — a privacy posture call (SC-10) that respects the deletion right (person unlinked) without losing the H3 corpus (no auto-purge). Synchronous capture can *visibly* (counted via `capture_error`) drop an event under failure rather than guaranteeing zero-loss; a durable outbox is the named growth path. The contract published here (`signals.capture.*`, `signals.selectors.*`, the two event shapes, soft `App.id`/`Tag.id` refs) is the stable cross-feature surface and is **additive-only** by design (a new event kind is a new enum value + recorder, never a change to these).

### D-6: Catalogued-app cross-feature contract — the accepted `catalog.App`, read through selectors only
- **Date:** 2026-06-17
- **Stage / feature:** `2-design` / `submission-intake` (Software Architect) — **approved 2026-06-17**; implemented in Stage 4 (`apps/catalog`). **Rollout ([DESIGN §11/§12](features/submission-intake/DESIGN.md)):** a downstream feature must adopt these rules *before* it stores any app reference.
- **Decision:** The **catalogued unit is an `accepted` `catalog.App`**, and its **stable cross-feature reference is `App.id` (UUID)** — never the URL, name, or any other field. Every downstream feature (**`app-pages`, `editorial-curation-tools`, `signal-capture`, `developer-dashboard`, the matcher/digest**) must: **store the `App.id`** and nothing else as the app handle; **read the catalog only through `selectors.list_catalogued_apps` / `get_catalogued_app`**, which return **accepted apps only** (a `pending`/`rejected`/`withdrawn` app is never presented as catalogued — never read `catalog_app` directly past this surface); **treat an app's tags as taxonomy `Tag.id`s under the [D-5](DECISIONS.md) contract** (resolve with `taxonomy.resolve_tag`, never store/compare by label); and **treat media as an ordered list of images** under the §9 slots/limits (1–8 files; PNG/JPEG/WebP; ≤5 MB) with stable order. The accepted-app JSON/selector shape (id, name, description, url, resolved tags, ordered media) is **additive-only**; new lifecycle is a new endpoint, never a change to these.
- **Why:** Every Phase-1 feature reads the catalog, so they must agree on *what* the catalogued unit is and *how* to reference it, or they will drift (read non-accepted apps as live, key signals to an unstable handle, or show stale tag labels). Routing all reads through two selectors makes the **accepted-only** guarantee (AC9) a single enforced boundary rather than a convention each consumer must re-implement. A UUID handle keeps references valid across URL/metadata edits; deferring to `resolve_tag` inherits D-5's rename/merge safety. This is global because **more than one later feature would be wrong to contradict it.**
- **Alternatives rejected:** (a) **Reference an app by URL or `normalized_url`** — both mutate on owner edit and duplicates legitimately coexist (rejected/withdrawn), so the handle would be unstable and ambiguous. (b) **Let consumers query `catalog_app` directly with their own status filter** — duplicates the accepted-only rule in every feature, the exact drift this contract prevents; one missed filter leaks a pending/withdrawn app into the catalog. (c) **Hard M2M FK from app to `taxonomy.Tag`** — couples schemas and fights `resolve_tag`'s read-time remap (the reason D-5 is a soft reference). Full rationale in [features/submission-intake/DESIGN.md](features/submission-intake/DESIGN.md) §11/§13.
- **Sacrifices / consequences:** Downstream takes a `resolve_tag` lookup per tag dereference (cached projection is the documented growth path, inherited from D-5). The selector surface is the stable cross-feature API and is additive-only by design; a new consumer need (e.g. anonymous public read for `app-pages`) is a one-line `AllowAny` endpoint over the same selector, noted not built (no speculative abstraction). The accepted-only guarantee lives in two selectors, so a consumer that bypasses them (reads `catalog_app` directly) can break the contract — flagged in DESIGN §11 so it is not done.

### D-5: Interest-taxonomy shape & cross-feature tag-reference contract
- **Date:** 2026-06-17
- **Stage / feature:** `2-design` / `interest-taxonomy` (Software Architect) — **approved (A5, 2026-06-17); implemented in Stage 4** (`apps/taxonomy`: `selectors.is_valid_tag`/`resolve_tag`/`list_active_tags`, `Tag.id` UUID reference). **Rollout (DESIGN §12):** a downstream feature must adopt these three rules *before* it stores any tag reference.
- **Decision:** The shared interest vocabulary is modeled as **flat tags + named clusters joined by a many-to-many membership** (no tag→tag hierarchy; a tag may belong to one *or more* clusters). The **stable cross-feature reference to a tag is its UUID `id`** — never its display label and never its (internal, immutable) `slug`. Every feature that records an interest/label (**`interest-profile`, `submission-intake`, the matcher, `editorial-curation-tools`**) must: **store the `Tag.id`**; **validate** supplied values at its write boundary with the taxonomy's `is_valid_tag(id)` (reject off-vocabulary input — no tag coining); and **resolve** at read with `resolve_tag(id)` (which transparently follows renames and retire/merge successors, and never drops a reference). **Clusters are first-class from day one** so that future **cluster-to-cluster adjacency** (ring-based expansion, vision §2.2) is a purely additive table over existing clusters — **no re-tagging, no destructive migration** (AC8). Tags are **retired (soft), never hard-deleted**, so stored references stay valid (reference-break-rate = 0).
- **Why:** Matching only works if both sides speak one language drawn from one closed dictionary (brief). A label-independent UUID reference is the only thing that keeps stored interests/app-labels valid across editorial renames and merges (R4, AC6/AC7). Making clusters first-class now is the cheap insurance against the Coordinator's named future change (rings) without over-building adjacency today (ITX-2, AC8). This is global because **more than one later feature would be wrong to contradict it** (store-by-label, coin off-vocabulary tags, or hard-delete tags would each break the substrate).
- **Alternatives rejected:** (a) **Single-cluster FK on each tag** — simpler, but the brief requires "one *or more* clusters" and it forecloses a tag spanning clusters. (b) **Arbitrary tag→tag hierarchy** — over-scope (R2); adjacency is a *cluster*-level relation, so a tag tree buys nothing AC8 needs and adds traversal cost. (c) **Hard-delete on retire / rewrite downstream references on merge** — both break stored references (reference-break-rate > 0) and the merge-rewrite touches tables this feature does not own; rejected for non-destructive read-time resolution. (d) **Reference tags by label/slug string** — defeats safe rename; rejected. Full rationale in [features/interest-taxonomy/DESIGN.md](features/interest-taxonomy/DESIGN.md) §7/§13.
- **Sacrifices / consequences:** Downstream features take a per-dereference `resolve_tag` lookup (vs caching) — accepted at MVP scale, with a cached projection as the documented growth path. The "≥1 cluster per active tag" invariant cannot be a single DB constraint, so it is enforced at the one write path + a `check_taxonomy` check rather than in the schema. No dedicated audit table ships here (lifecycle state on rows + observability + Django-admin LogEntry); a rich audit lands with `editorial-curation-tools`. The contract published here (`Tag.id`, `is_valid_tag`, `resolve_tag`, `list_active_tags`, cluster membership) is the stable cross-feature surface and is additive-only by design.

### D-4: Tech stack — Django + DRF + PostgreSQL; shared-code root `apps/`
- **Date:** 2026-06-14
- **Stage / feature:** `2-design` / `identity-accounts` (Software Architect)
- **Decision:** The platform is built on **Python 3.12+ / Django 5.x** with **Django REST Framework** for JSON APIs and **PostgreSQL 15+** as the datastore (Django ORM + migrations). Authentication is **passwordless email magic-link** over **Django server-side sessions**; authorization uses **Django `Group`s as roles** enforced through a single permission point. Email delivery is a **pluggable interface** (`apps/core/email.py`) whose concrete provider is ops/env config, not code. The **shared-code root is `apps/`**: every feature is a Django app under it, with cross-cutting reusable code in `apps/core/` and the canonical account/role surface in `apps/accounts/`. For this feature the UI is **server-rendered Django templates**; a richer SPA frontend is deferred until a surface needs one (not chosen here).
- **Why:** Chosen by the user for Python familiarity and as the strongest base for the later **data/ML-heavy** internal engine (Quality-Score pipeline, matching/ring computation, integrity graph analysis — all §3/§4/§6 of the vision), which lives most naturally in Python. Django's batteries-included auth, sessions, groups/permissions, ORM, migrations, and admin minimize hand-rolled, security-sensitive code (CLAUDE.md §5.4 fail-loud, §5.5 boring-and-well-understood). PostgreSQL covers relational identity now and the relational/analytic needs of later features.
- **Alternatives rejected:** (a) **TypeScript full-stack** (Next.js + Auth.js + Postgres) — viable and lower-wiring for the web surfaces, but the user preferred Python and JS is a weaker base for the ML/graph internal engine. (b) **Framework-light TS API** (Fastify + separate SPA) — cleanest boundaries but requires hand-built auth/session glue (more security surface to get wrong) for no MVP benefit. Both logged so a later feature does not re-litigate the stack.
- **Sacrifices / consequences:** Two languages once a rich SPA frontend is added (Python backend + JS frontend); accepted because identity-accounts needs no SPA yet. Django's monolith-first shape means we accept a **modular monolith** now — the documented growth path is extracting services at scale, a deliberate bounded trade-off (D-2 / CLAUDE.md §5.2). The account+role contract published here (`Account.id`, `Account.email`, `HasRole`/`require_role`, role constants) is the stable cross-feature API every later feature builds on; it is additive-only by design.

### D-3: Identity model — one account, one access method, role-based authorization
- **Date:** 2026-06-14 *(revised 2026-06-14 per user A1 direction — generalized from "dual capability (reader + developer)" to an extensible role model; no downstream feature had consumed the prior wording yet)*
- **Stage / feature:** `1-define` / `identity-accounts` (Product Analyst)
- **Decision:** A person has a **single account** and a **single way to sign in**, regardless of what they are allowed to do. What an account may do is governed by the **roles** it holds. The MVP defines three roles — **user** (the base role every account has: discover apps, receive/open the curated digest), **developer** (submit and own apps), and **admin** (privileged internal/editorial/operations actions) — and the model is **extensible** so future roles can be added without a new access method or a redesign of authentication. An account may hold more than one role (e.g. user + developer). The **developer** role is self-serve (any account may take it — DL-1); the **admin** role is **not** self-serve and is granted only to authorized staff.
- **Why:** The beachhead niche is vibecoded webapps (D-1), where the same solo/tiny-team person is routinely both a discoverer and a builder, so one identity avoids fragmented logins and keeps a person's history unified. Modelling permitted actions as roles (rather than a fixed reader/developer pair) is the simpler, more scalable choice (CLAUDE.md §5.2): the platform already needs privileged editorial/operations access, and a single extensible role system absorbs that — and any later role — without a second identity system or a separate admin login.
- **Alternatives rejected:** (a) Separate user and developer accounts — forces the niche's core persona to maintain two identities and splits their signal/history. (b) A fixed two-capability flag (the prior D-3 wording) — would have required a *second* identity/access mechanism the first time the platform needed admins or any third role, contradicting "design for change" (§5.2). (c) A separate out-of-band admin login — duplicates the auth surface and the account model for no benefit.
- **Sacrifices / consequences:** Cross-feature contract: `submission-intake`, `signal-capture`, `interest-profile`, `developer-dashboard`, and `editorial-curation-tools` must key off one account whose permitted actions are gated on **roles** (not on a separate account type or login). Every privileged surface authenticates through this one feature and checks an `admin` role; `editorial-curation-tools` builds the admin *tooling*, not a parallel identity system. Role *assignment* mechanics (how a role is granted, especially admin) are a Stage-2 design concern. Team/organization accounts remain out of scope at MVP. The added generality is a deliberate, bounded cost justified by an already-known need (admin) plus a named likely change (future roles), not speculation (§5.5).

### D-2: No hard launch constraints at MVP — start small, scale as we go
- **Date:** 2026-06-14
- **Stage / feature:** `0-coordinator` (resolving CONTROL.md D3)
- **Decision:** No fixed deadline, budget, platform, or compliance/privacy ceiling is imposed up front. The guiding posture is to build the smallest honest slice first and grow capacity (data, users, infra) incrementally.
- **Why:** The MVP's job is to validate H1–H3 cheaply (breakdown §1). Premature hard constraints would over-specify non-functional targets before we have real load or a chosen stack.
- **Alternatives rejected:** Fixing a launch deadline / infra budget now — rejected because there is nothing yet to size against; it would be guesswork that later features would be wrong to treat as binding.
- **Sacrifices / consequences:** Non-functional targets (scale, latency, retention/consent specifics) are **deferred to each feature's Stage 1–2**, not given globally. "Scale as we go" is a posture, **not** a licence to ship hacks — CLAUDE.md §5.2 (designs must still work at 100×, or document the bounded trade-off) still binds. Privacy/compliance posture for behavioral data remains an explicit open design fork (breakdown §7 Q4), now un-gated but unresolved.

### D-1: Beachhead niche is "vibecoded webapps"
- **Date:** 2026-06-14
- **Stage / feature:** `0-coordinator` (resolving CONTROL.md D1)
- **Decision:** The single launch vertical is **vibecoded webapps** — small web applications built rapidly (often AI-assisted / "vibe coding") by solo developers and tiny teams.
- **Why:** The design mandates a sequenced narrow launch with one beachhead niche (vision §5.4, breakdown §2). This niche is rich in exactly the platform's target persona — unknown solo devs with $0 marketing shipping real apps — making it the cheapest honest test of the one-line north star (breakdown §1).
- **Alternatives rejected:** Indie productivity tools / indie games on a single platform (the breakdown's example niches) — rejected in favour of vibecoded webapps because web apps need no app-store gatekeeper or per-platform install flow, lowering the bar for both submission and the cross-platform attribution prototype.
- **Sacrifices / consequences:** Scopes the **content**, not the component structure (components are niche-agnostic, breakdown §6). Directly shapes `interest-taxonomy` (tag vocabulary), `submission-intake` (what "an app" is + the objective gate), and the founding catalog. Web-only at MVP means native-install attribution is out of scope for now; if the platform later expands beyond web, `signal-capture` attribution must be revisited.
